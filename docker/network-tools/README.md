# network-tools

## Описание

В этой задаче внутри Docker-контейнера были изучены и проверены сетевые команды:

- `curl`
- `ping`
- `ss -lntp`
- `ip a`
- `hostname`
- `nslookup`

Также был поднят локальный HTTP-сервер через:

```sh
python3 -m http.server 8000
```

После этого было проверено, что порт `8000` слушается и приложение отвечает.

---

## Проверка hostname

Команда:

```sh
hostname
```

Вывод:

```text
4ea981a00fa8
```

Команда `hostname` показала имя контейнера внутри Linux-среды. Это подтверждает, что команды выполнялись внутри контейнера.

---

## Проверка сетевых интерфейсов через ip a

Команда:

```sh
ip a
```

Важная часть вывода:

```text
11: eth0@if35: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 65535 qdisc noqueue state UP group default
    inet 172.17.0.4/16 brd 172.17.255.255 scope global eth0
```

Команда `ip a` показала, что у контейнера есть сетевой интерфейс `eth0` и IP-адрес `172.17.0.4` внутри Docker-сети.

---

## Проверка открытого порта через ss -lntp

Команда:

```sh
ss -lntp
```

Вывод:

```text
State         Recv-Q        Send-Q               Local Address:Port               Peer Address:Port       Process
LISTEN        0             5                          0.0.0.0:8000                    0.0.0.0:*           users:(("python3",pid=1,fd=3))
```

Команда `ss -lntp` показала, что порт `8000` открыт и слушается процессом `python3`.

## Проверка ответа локального HTTP-сервера

Команда:

```sh
curl http://localhost:8000
```

Вывод:

```text
network-tools container is working
```

Эта проверка показала, что HTTP-сервер внутри контейнера не просто запущен, а реально отвечает на запрос.

## Проверка внешних сайтов через curl -I

### example.com

Команда:

```sh
curl -I https://example.com
```

Вывод:

```text
HTTP/2 200
date: Tue, 28 Apr 2026 18:12:04 GMT
content-type: text/html
server: cloudflare
```

Код `HTTP/2 200` означает, что контейнер смог обратиться к внешнему сайту и получил успешный HTTP-ответ.

### github.com

Команда:

```sh
curl -I https://github.com
```

Вывод:

```text
HTTP/2 200
date: Tue, 28 Apr 2026 18:46:08 GMT
content-type: text/html; charset=utf-8
server: github.com
```

Код `HTTP/2 200` подтверждает, что второй внешний сайт тоже доступен из контейнера.

`curl` проверяет именно HTTP/HTTPS-ответ сайта или приложения.

## Проверка DNS через nslookup

Команда:

```sh
nslookup google.com
```

Вывод:

```text
Server:         192.168.65.7
Address:        192.168.65.7#53

Non-authoritative answer:
Name:   google.com
Address: 172.217.19.238
```

`nslookup` проверил работу DNS.

DNS преобразует доменное имя в IP-адрес:

```text
google.com -> 172.217.19.238
```

Вывод показывает, что контейнер смог получить IP-адрес для домена `google.com`, значит DNS внутри контейнера работает.

## Чем отличаются команды и зачем нужна каждая

В этой задаче команды проверяют разные части сетевой работы контейнера.

### `hostname`

Показывает имя контейнера внутри Linux-среды.

Назначение: понять, что команды выполняются именно внутри контейнера, а не на хостовой машине.

---

### `ip a`

Показывает сетевые интерфейсы и IP-адреса контейнера.

Назначение: убедиться, что контейнер подключён к Docker-сети и получил внутренний IP-адрес.

В этой проверке было видно:

```text
eth0
172.17.0.4
```

Это значит, что у контейнера есть сетевой интерфейс `eth0` и IP-адрес внутри Docker-сети.

---

### `ss -lntp`

Показывает открытые TCP-порты и процессы, которые их слушают.

Назначение: убедиться, что внутри контейнера реально открыт порт `8000`.

В выводе было видно:

```text
0.0.0.0:8000
users:(("python3",pid=1,fd=3))
```

Это значит, что порт `8000` слушает процесс `python3`.

---

### `curl http://localhost:8000`

Проверяет HTTP-ответ локального приложения внутри контейнера.

Назначение: убедиться, что сервер не просто запущен и слушает порт, а реально отвечает на запрос.

Разница:

- `ss -lntp` показывает, что порт открыт;
- `curl http://localhost:8000` показывает, что приложение отвечает.

---

### `curl -I`

Проверяет HTTP/HTTPS-ответ внешнего сайта.

Назначение: убедиться, что контейнер может обращаться к сайтам в интернете и получать от них HTTP-ответ.

Флаг `-I` означает, что запрашиваются только заголовки ответа, без полной HTML-страницы.

Главное в выводе:

```text
HTTP/2 200
```

Код `200` означает успешный ответ от сайта.

---

### `ping`

Проверяет сетевой путь до внешнего адреса и время ответа.

Назначение: понять, есть ли сетевой маршрут от контейнера до внешнего хоста.

Главная часть вывода:

```text
3 packets transmitted, 3 received, 0% packet loss
```

Это значит, что контейнер отправил 3 пакета и получил 3 ответа. Потерь нет.

Также `ping` показывает задержку:

```text
time=3.30 ms
time=1.54 ms
time=1.46 ms
```

Важно: `ping` не проверяет работу сайта как веб-сервиса. Он проверяет именно сетевой путь и задержку.

---

### `nslookup`

Проверяет DNS.

Назначение: убедиться, что контейнер может преобразовывать доменные имена в IP-адреса.

Пример:

```text
google.com -> 172.217.19.238
```

Если DNS не работает, контейнер может не понимать доменные имена вроде `google.com`, даже если сама сеть есть.

---

## Общее заключение

Эти команды проверяют разные уровни работы сети:

- `hostname` — показывает, где мы находимся;
- `ip a` — показывает, есть ли у контейнера сетевой интерфейс и IP-адрес;
- `ss -lntp` — показывает, слушается ли нужный порт;
- `curl http://localhost:8000` — проверяет ответ локального HTTP-сервера;
- `curl -I` — проверяет HTTP/HTTPS-ответ внешнего сайта;
- `ping` — проверяет сетевой путь до адреса и задержку;
- `nslookup` — проверяет DNS, то есть преобразование имени сайта в IP-адрес.

Главная разница между `ping` и `curl`:

- `ping` отвечает на вопрос: есть ли сетевой путь до адреса;
- `curl` отвечает на вопрос: отвечает ли HTTP/HTTPS-сервис.

Поэтому одной команды недостаточно. Вместе они показывают полную картину: контейнер подключён к сети, умеет находить домены, ходить во внешний интернет, слушать порт `8000` и отдавать ответ через локальный HTTP-сервер.
